## Current state

What works:

- The full three-layer vision is live: one shared prioritization engine with a
  core loop, a guided-process tour, and an advanced method deep-dive over it.
- Set targets, edit the landscape (features/cost/locks), switch objective
  (min-set vs max-coverage under budget), add a compactness penalty, weight
  features, run the guided tour, and compare greedy against the exact optimum
  (at default targets: greedy 117 vs exact 101, a 15.8% gap).
- 38 tests pass; engine coverage 100% lines / ~88% branches (gate is 80%). CI is
  green and deploy is automated. Zero open issues; only `main` exists.

Architecture:

- `src/engine/` pure, headless, unit-tested: types, amounts, feasibility,
  attainment, greedy (with options: objective/weights/boundary), exact (MILP),
  compare, constants.
- `src/data/scenario.ts` the synthetic landscape generator, `WorkingUnit` editing
  model, `toProblemFromUnits`, and `NEIGHBORS` adjacency.
- `src/ui/` React: App, GridView, CostTargetCurve, TourPanel, tour data, color.
- The exact solver is a lazy dynamic import (separate ~52 kB chunk).

Process notes / gotchas:

- CI runs `format:check` as its own gate. Run the full gate (format, lint,
  typecheck, test, build) right before committing. Two self-inflicted slips this
  session, both caught before merge: committed to local `main` once before
  branching (moved to a branch, reset main); pushed once before re-running
  Prettier (CI caught it). Re-run format after any edit that follows your last
  Prettier run.
- The new `react-hooks` lint rules reject ref-factory callbacks and
  set-state-in-effect. Patterns used instead: a single root ref + `data-region`
  attributes queried in an effect (tour), and clearing derived state in the
  input handlers rather than an effect (comparison).
- Browser verification uses Playwright installed with `--no-save`; a plain
  `npm install`/`uninstall` prunes it, so reinstall before screenshotting. Run
  the dev server as a real background job, not inline `&`.
- No em dashes anywhere (code, docs, commits, UI) per the working rules.

Known limits / simplifications (by design):

- Synthetic data only; abstract grid (no GIS/map). Greedy is not optimal (that is
  the teaching point the exact panel makes). Exact solver is bounded to the small
  grid. All values illustrative.

## Risks or decisions needed

- Real data ingestion is the largest open design question and should get an ADR
  first (source, crosswalk, offline preprocessing); it likely gates elevation,
  scaling, and any "real" credibility for the tool.
- Scaling past ~30x30 would push grid rendering from SVG toward canvas and the
  edit tool toward a brush; revisit before committing to larger landscapes.
- Solver ADR was updated mid-session: the exact solver is javascript-lp-solver,
  not glpk.js. If a future need requires a stronger solver, revisit that ADR.
