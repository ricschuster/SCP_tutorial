## Completed

Shipped (all merged to `main`):

- #55 (closes #53): swapped the exact minimum-set solver from glpk.js to **HiGHS**
  at a **1% relative MIP gap**. With 8 features GLPK could no longer prove
  optimality in an interactive budget (~10s, hit its time limit). Benchmarking
  showed a mipgap alone does not fix GLPK (it still hits the 10s wall at low
  targets), while HiGHS at a 1% gap stays under ~1.2s across the whole target range
  with at most ~0.7% objective deviation. HiGHS ships only a synchronous wasm
  build, so it runs in a **Web Worker we own** (`src/engine/highs.worker.ts` +
  `highs-worker-client.ts`), with a synchronous fallback under Vitest
  (`highs-sync.ts`); `highs-runner.ts` picks the path by `typeof Worker`. The UI is
  relabelled "near-optimal optimum (within 1%)". Dropped glpk.js and its Vitest
  alias. ADR `docs/decisions/2026-07-19_exact-solver-highs-at-scale.md` (supersedes
  the 2026-07-18 revisit ADR) with the full benchmark.
- #56: the guided tour's greedy-vs-near-optimal step now **auto-runs** the solve on
  arrival (a `TourStep.compute` flag drives `runCompare`) instead of pointing the
  learner at the button, now that the solve is ~0.4s not ~10s.
- #57: **exact max-coverage comparison** on the Method tab. The compare panel now
  respects the objective toggle: under max coverage it compares greedy against a
  near-optimal max-coverage optimum (`solveExactMaxCoverage`) for the current budget
  and weights, reporting coverage as a percentage of the achievable ceiling with the
  gap in percentage points. New engine helpers `coverageValue` and `compareCoverage`.

Decisions made with the user:

- #53 direction: after benchmarking, switch to HiGHS (not just add a mipgap to
  GLPK), because a mipgap leaves GLPK slow at low targets while HiGHS is robust.
- Run HiGHS in a Web Worker (keep the solve off the main thread), and relabel the
  reference as "near-optimal (within 1%)" rather than "exact".
- Max coverage solves to the **true optimum (gap 0)**, not the 1% gap min-set uses:
  it is cheap (~0.1-0.2s, ~1.4s worst case), and a gap would let the "optimum" come
  back below greedy at generous budgets (where greedy already reaches the ceiling),
  showing a confusing negative gap. "Within 1%" still holds since 0 <= 1%.

## Current state

What works:

- Explore: set per-species targets for 8 species, paint one of six covers, watch
  habitat and cost derive and the solver re-solve live; inspect any cell; toggle
  irreplaceability; lock in/out. Combined-habitat scalar map + species spotlight;
  feature table.
- Method: switch objective (min-set / max-coverage), add compactness and
  connectivity penalties (with an optional same-cover boost), weight species, read
  the cost-target curve, and compare greedy against a near-optimal optimum. The
  compare panel now covers **both** objectives: min-set reports a cost gap;
  max-coverage reports a coverage gap (percent of the achievable ceiling, gap in
  points) within the budget.
- Guided tour: short (6) + optional full (14) narrative that drives the real UI,
  now auto-running the compare step, and restoring state on close.
- Share/bookmark the full scenario via the URL hash.
- 98 tests pass; engine coverage ~98% (gate is 80%). Every PR auto-merged after CI.

Architecture (updated this session):

- `src/engine/` pure: types, greedy (weights, boundary, connectivity), attainment
  (now with `coverageValue`), compare (`compareSolutions` + `compareCoverage`,
  sharing a `SelectionDiff`), feasibility, irreplaceability, constants, and the
  exact solver.
- Exact solver: `exact.ts` builds the MILP as a CPLEX-LP string (`buildMinSetLp`,
  `buildMaxCoverageLp`) and delegates solving to `runHighs`. `solveExact` (min-set,
  1% gap) and `solveExactMaxCoverage` (max-coverage, gap 0). HiGHS runs in a Web
  Worker in the browser and synchronously under Vitest. The two browser-only worker
  files are excluded from the coverage gate and verified in a real browser instead.
- `src/data/`: land-cover, scenario, connectivity, share (codec).
- `src/ui/`: App (tabs, brush, inspector, tour driver, compare panel now
  objective-tagged and cleared on objective/budget/weight change), GridView
  (canvas), grid-draw, CostTargetCurve, TourPanel, tour data, color.

Known limits / simplifications:

- The reference solution is "near-optimal (within 1%)" for min-set, not proven to
  the cent (max-coverage is exact). The UI says so. In practice min-set greedy is
  clearly worse than the optimum, so the displayed gap is positive; a near-zero
  negative min-set gap is theoretically possible if greedy happens to be optimal.
- The HiGHS wasm chunk is ~1.2 MB gzipped, but it is dynamically imported (loaded
  only when the learner runs the compare), so the initial bundle is unaffected.
- Synthetic data; greedy is not optimal (the teaching point); the landscape is not
  a real Canadian ecoregion.
- `vite.config.ts` keeps `testTimeout: 20000` for the full-app UI tests (heavy in
  jsdom on CI).
